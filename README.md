# مالية — Maliya v2.0.0

تطبيق مالي شخصي عربي/إنجليزي (RTL أولًا)، يعمل محليًا بالكامل ويتزامن اختياريًا عبر Supabase، وقابل للتثبيت كتطبيق (PWA) على الجوال والكمبيوتر.

**الإصدار: 2.0.0 · مخطط البيانات: v8** — يرحّل تلقائيًا أي نسخة من v1 حتى v8.

## بنية المشروع
```
index.html            الهيكل
css/app.css           الأنماط
js/config.js          إعداد Supabase (لا يُرفع للمستودع — انظري .gitignore)
js/config.example.js  مثال الإعداد (هذا الذي يُرفع)
js/core.js  js/app.js وحدات التطبيق
manifest.json  service-worker.js  assets/   ملفات PWA والأيقونات
supabase/schema.sql  supabase/policies.sql  إعداد قاعدة البيانات وسياسات RLS
```

## 1) تشغيل محلي فورًا (بدون سحابة)
أي خادم ثابت يكفي:
```
python3 -m http.server 8080
# ثم افتحي http://localhost:8080
```
اتركي `js/config.js` كما هو (قيم `YOUR-…`) والتطبيق يعمل محليًا 100٪ ويعرض في «المزيد» بطاقة توضّح أن السحابة غير مكوّنة.

## 2) ربط Supabase (المزامنة بين أجهزتك)
1. أنشئي مشروعًا في supabase.com.
2. من **SQL Editor** شغّلي `supabase/schema.sql` ثم `supabase/policies.sql` بهذا الترتيب.
3. من **Project Settings → API** انسخي:
   - **Project URL**
   - **publishable (anon) key** — العام فقط. ⚠️ لا تضعي `service_role` في الواجهة أبدًا؛ سياسات RLS المرفقة تفعّل العزل الكامل: كل مستخدم يقرأ ويكتب صفوفه فقط (`auth.uid() = user_id`).
4. انسخي `js/config.example.js` إلى `js/config.js` وضعي القيمتين في:
   - `SUPABASE_URL`
   - `SUPABASE_PUBLISHABLE_KEY`
5. أعيدي فتح التطبيق → «المزيد» → **تسجيل الدخول** (إنشاء حساب بالبريد وكلمة المرور، مع استعادة كلمة المرور).

## 3) ترحيل بياناتك الحالية (من نسخة المرحلة 7)
1. في النسخة القديمة: «المزيد» → **تصدير نسخة JSON** واحتفظي بالملف.
2. في النشر الجديد (رابط GitHub Pages أصل مختلف، فالتخزين المحلي لا ينتقل وحده): «المزيد» → **استيراد نسخة** واختاري الملف — يترحّل تلقائيًا إلى v8 مع تحقق كامل.
3. سجّلي الدخول → سيسألك التطبيق **مرة واحدة** «رفع بياناتك الحالية للسحابة؟» — قبل الرفع تُحفظ نسخة محلية (`maliya.premigration.v1`)، وبعده تُقارن الأعداد والأرصدة؛ أي فشل في التحقق = تراجع تلقائي وبياناتك المحلية سليمة. تكرار الزر لا يرفع نسخة ثانية أبدًا.

## 4) كيف تعمل المزامنة (موثّق)
- **مستند حالة واحد لكل مستخدم** في `maliya_state` مع ختم `updated_at` كرمز تفاؤلي (CAS).
- كل حفظ محلي يعلّم الحالة «متغيّرة» ويدفع تلقائيًا بعد ثوانٍ عند الاتصال، مع زر **مزامنة الآن** وطابع «آخر مزامنة».
- الحالات الظاهرة: متزامنة ✓ / جاري المزامنة / غير متصلة / بانتظار الترحيل / خطأ / تعارض.
- **أوفلاين**: التطبيق يفتح ويعمل بالكامل (Service Worker + التخزين المحلي)؛ أي دفعة أو تراجع أثناء الانقطاع يُدفع مرة واحدة فقط عند عودة الشبكة — إعادة المحاولة آمنة لأن الدفع «لقطة كاملة»، فالتكرار مستحيل بنيويًا.
- **التعارض** (تعديلات محلية + سحابة أحدث): لا نكتب فوق الأحدث بصمت أبدًا — تظهر لك ورقة اختيار، والنسخة الخاسرة تُحفظ احتياطيًا محليًا (`maliya.conflictbackup.v1`).

## 5) النشر على GitHub Pages
```
git init && git add . && git commit -m "Maliya v2.0.0"
git branch -M main
git remote add origin https://github.com/USERNAME/maliya.git
git push -u origin main
```
ثم: **Settings → Pages → Deploy from a branch → main / (root)**.
كل المسارات نسبية (`./`) و`manifest.json` يستخدم `start_url: "./"` — يعمل تلقائيًا تحت مسار المستودع الفرعي `username.github.io/maliya/`. ملاحظة: `js/config.js` مستثنى من Git؛ بعد أول نشر ارفعيه يدويًا أو أضيفيه عبر واجهة GitHub (المفتاح publishable عام بطبيعته والحماية عبر RLS).

## 6) التثبيت كتطبيق (PWA)
- **آيفون**: سفاري → مشاركة → **إضافة إلى الشاشة الرئيسية**.
- **أندرويد/كروم**: قائمة المتصفح → **تثبيت التطبيق**.
عند نشر تحديث، يظهر إشعار «تحديث جاهز» — أغلقي التطبيق وافتحيه. عامل الخدمة لا يخزّن أبدًا أي طلب لـ Supabase (مصادقة أو بيانات).

## 7) النسخ الاحتياطية
- تصدير/استيراد JSON كامل (متوافق مع كل النسخ من v1).
- **نسخة احتياطية سحابية الآن** تكتب لقطة في `maliya_backups`.
- نسخ طوارئ محلية تلقائية: قبل الترحيل الأول وعند حسم أي تعارض.

## نسخة الطوارئ الموحّدة
`standalone/maliya-standalone.html` — الملف الكامل بذاته يعمل بفتحه مباشرة (محليًا بدون سحابة، ويمكن تفعيلها بتحرير كتلة `MALIYA_CONFIG` داخله).
